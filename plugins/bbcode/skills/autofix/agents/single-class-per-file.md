# Sub-Agent: single-class-per-file

## Purpose

Enforce **one top-level declaration per file** for classes, interfaces, enums, and (where appropriate) types. Files that bundle multiple declarations must be split, with each declaration moved into its own correctly-named file under the `/<domain>/<type>/<file>` structure.

This sub-agent fixes one of the most common code smells: "junk drawer" files that grow into untyped grab-bags.

## Detection

You will be given a target file list. Flag a file for splitting if **any** of the following apply.

### Code-Smell Filenames (instant flag)

These plural / generic names are almost always grab-bags. Always inspect them:

- `interfaces.*` / `interface.*`
- `models.*` / `model.*`
- `types.*` / `type.*`
- `constants.*` / `constant.*`
- `enums.*` / `enum.*`
- `utils.*` / `util.*` / `helpers.*` / `helper.*`
- `common.*` / `shared.*` / `misc.*`
- `index.*` **only if** it contains declarations rather than just re-exports

A code-smell filename is a **strong signal but not proof** — confirm by reading the file. If it really does contain just one declaration, leave it alone and recommend a rename instead (see Fix → Rename-only case).

### Content-Based Detection (always apply)

Regardless of filename, flag a file that contains **more than one** of:
- `class` declarations (top-level)
- `interface` declarations (top-level)
- `enum` declarations (top-level)
- `type` aliases that represent a distinct domain concept (judgement call — small union helpers tied to one class can stay)

Ignore:
- Type-augmentation / module-augmentation files (`*.d.ts` with `declare module`)
- Generated files (look for header comments like "AUTOGENERATED", "DO NOT EDIT")

Test files are **not** auto-ignored — see the "Test layout" step below.

## Fix

### Step 0 — Detect the project's test layout (do this once, before any fixes)

Before moving or renaming anything, figure out **how this project structures tests**. Inspect the repo and pick the best match:

- **Co-located** — test files sit next to their source: `src/auth/user.ts` + `src/auth/user.test.ts` (or `.spec.ts`, `_test.go`, `_spec.rb`, etc.)
- **Mirrored separate tree** — tests live in a parallel directory rooted elsewhere, with the **same internal structure** as source: `src/auth/user.ts` + `tests/auth/user.test.ts` or `test/auth/user_test.py`
- **Sibling `__tests__` folders** — tests live in a `__tests__/` folder next to source: `src/auth/user.ts` + `src/auth/__tests__/user.test.ts`
- **No tests / cannot determine** — proceed without test propagation, but note it in the output

How to detect:
1. Pick a known source file in the target list. Search the repo for files whose name matches `<basename>.test.*`, `<basename>.spec.*`, `<basename>_test.*`, `test_<basename>.*`, etc.
2. Compare paths: same directory → co-located; different root with mirrored subpath → mirrored; inside a sibling `__tests__/` → sibling folder.
3. If multiple patterns coexist, pick the dominant one for the area you're editing (don't impose a global convention on a project that mixes styles).

**Record the detected layout** — every fix below must propagate to test files using this layout.

### Standard split

1. **Pick a target directory** following `/<domain>/<type>/<file>`:
   - `domain` = inferred from the file's current location (keep it where it lives unless clearly wrong)
   - `type` = chosen to match the declaration kind. The list below is **non-exhaustive** — other valid types include `store`, `client`, `factory`, `adapter`, `provider`, `middleware`, etc. Prefer whatever convention the project already uses; only invent a new type segment if nothing existing fits.
     - **`model`** — for `class`, `interface`, `enum`, and domain `type` aliases (interfaces and enums are part of the model domain type, not their own segment)
     - **`service`** — for service classes
     - **`repository`** — for repository classes
     - **`controller`** / **`handler`** — request entry points
     - **`store`** — state stores
     - **`client`** — external API / SDK clients
     - **…** — any other type segment that fits the declaration's role and matches the project's convention
   - For languages that require all package files in the same folder (Go, etc.), skip the `<type>` level and use `/<domain>/<file>`
2. **Create one file per declaration**, named in the project's existing case convention (kebab-case for TS/JS files in this repo unless the surrounding files use another style — match what's already there).
3. **Move** each declaration into its own file. Carry along:
   - Imports it actually uses (drop unused ones)
   - JSDoc / docstring / comments attached to the declaration
   - Any tightly-coupled private helper used by exactly one declaration (move it with the declaration that uses it)
4. **Update every importer** of the original file. Use repo-wide search to find all imports and rewrite them to point at the new file paths. Do not leave the original file as a re-export shim — delete it. (Backwards-compatibility shims are explicitly disallowed by the project conventions.)
5. **Propagate to tests** using the layout detected in Step 0:
   - **Co-located**: when source moves from `src/auth/interfaces.ts` to `src/auth/model/user-identity.ts`, move the test from `src/auth/interfaces.test.ts` to `src/auth/model/user-identity.test.ts`. If the original test file covers multiple declarations, split it the same way the source was split — one test file per new source file, carrying the relevant tests with their declaration.
   - **Mirrored separate tree**: same logic, but rebase paths onto the test root. `src/auth/interfaces.ts` → `src/auth/model/user-identity.ts` implies `tests/auth/interfaces.test.ts` → `tests/auth/model/user-identity.test.ts`.
   - **Sibling `__tests__` folders**: keep tests in the matching `__tests__/` folder next to the new source file.
   - Update test file imports to point at the new source paths. Do not change test logic.
   - If no test file exists for a given source file, skip — do not invent tests.
6. **Add a doc comment** at the top of each new declaration if one is missing, in the language's standard format (JSDoc for TS/JS, godoc for Go, docstrings for Python, etc.).

### Rename-only case

If a code-smell filename contains exactly **one** declaration, do not split — instead, rename the file to match the declaration and update all importers. Example: `interfaces.ts` containing only `UserRepository` → rename to `user-repository.ts` (keeping it under the same directory unless the directory is itself a code smell like `interfaces/`). Apply the same rename to the corresponding test file using the detected test layout.

### Code-smell directory case

If a directory is itself a grab-bag (`interfaces/`, `models/`, `types/`, `constants/`) **and** is not the `<type>` segment of a `/<domain>/<type>/` structure, flag it in the report but do not auto-restructure the directory tree — that's a larger refactor that needs human judgement. Just note it in your summary.

## Constraints

- **Do not change behavior.** This is a structural refactor only. No logic edits.
- **Do not break imports.** Every existing importer must end up pointing at the new location. Run a grep across the whole repo for the old module path before declaring the file done.
- **Do not introduce re-export barrels** (`index.ts` files that re-export everything) unless one already existed for that directory. Match the surrounding convention.
- **Keep tests in sync** with source moves using the layout detected in Step 0 — never leave a renamed source file with its test file pointing at a stale path. But do not rewrite test *logic*; only move/rename test files and update their imports.
- **Skip generated / vendored code** entirely.
- **Match existing style.** If the project uses `PascalCase.ts` filenames, use that. If it uses `kebab-case.ts`, use that. Look at neighbouring files to decide.
- If you are uncertain whether a split is safe (e.g. circular-dependency risk, declarations share private state), **leave the file untouched and report it** rather than guessing.

## Output

After processing all flagged files, return a structured summary:

```

Co-located (e.g. src/auth/user.ts + src/auth/user.test.ts)

### Files split
- src/auth/interfaces.ts → src/auth/model/user-identity.ts, src/auth/model/session-token.ts
  tests: src/auth/interfaces.test.ts → src/auth/model/user-identity.test.ts, src/auth/model/session-token.test.ts
- ...

### Files renamed
- src/billing/models.ts → src/billing/model/invoice.ts
  tests: src/billing/models.test.ts → src/billing/model/invoice.test.ts
- ...

### Files flagged but skipped
- src/legacy/grab-bag.ts — circular import risk between Foo and Bar; needs human review

### Directories flagged (no auto-fix)
- src/common/interfaces/ — grab-bag directory, recommend restructuring under domain/<type>/
```
